# 问题记录：英文秒出、中文不出 / 很慢

> 状态：已分析根因，代码侧已做「共享 Audio + 多源回退」；中文链路仍依赖第三方在线接口，稳定性弱于英文。  
> 关联代码：`src/lib/tts.ts`  
> 关联文档：`docs/bug-手机点击无声音.md`

## 1. 现象

| 操作 | 常见结果 |
|------|----------|
| 点「英」 | 几乎立刻出声 |
| 点「中」 | 无声、转圈很久、或报错 |
| 听写（念中文） | 失败率明显高于英文跟读 |
| 报错原文（典型） | `在线发音失败（源1超时；play() can only be initiated by a user gesture.）` + `当前浏览器无系统语音接口` |

同一台手机、同一网络、同一页面上，**中英文表现不一致**。

---

## 2. 结论（先看这句）

**不是「中文按钮坏了」，而是中英文走了两套完全不同的在线发音源：**

- **英文** → 有道词典「读单词」接口：专为英文词条录音/合成，CDN 快、响应短、几乎一请求就能播。  
- **中文** → 百度等「任意文本 TTS」接口：要现场合成整句中文，更慢、更容易被限流/鉴权/跨域策略影响；第一个源一超时，旧逻辑再 `play` 会丢掉用户手势，于是中文整段失败。

系统 `speechSynthesis` 在多数手机（尤其微信内置浏览器）上**不可用**，无法给中文兜底，所以中文一旦在线失败就「完全不行」，而英文在线经常一次成功。

---

## 3. 代码里中英文差异（证据）

`src/lib/tts.ts` 中 `onlineUrls()`：

```text
英文 en-US：
  1. https://dict.youdao.com/dictvoice?audio={词}&type=2   （美音）
  2. https://dict.youdao.com/dictvoice?audio={词}&type=1   （英音）

中文 zh-CN：
  1. https://fanyi.baidu.com/gettts?lan=zh&text={释义}&...
  2. https://tts.baidu.com/text2audio?...
  3. 百度 text2audio 另一组参数
  4. https://dict.youdao.com/dictvoice?audio={中文}&le=zh   （对中文释义并不总是可靠）
```

| 维度 | 英文（有道 dictvoice） | 中文（百度 TTS 等） |
|------|------------------------|---------------------|
| 产品定位 | 词典「读这个英文词」 | 通用「读这段中文」 |
| 内容类型 | 多为预置/词典音频，短 | 实时 TTS，延迟更高 |
| 首包速度 | 通常很快 → 体感「秒出」 | 常需数百 ms～数秒；差网易超时 |
| 接口稳定性 | 教育站常用，相对稳 | 非官方开放 API，参数/风控会变 |
| 失败后影响 | 很少进「换源 + 超时」 | 源1超时 → 换源 → 易触发手势限制 |

## 4. 为什么会叠加上「手势 / 无系统语音」问题

完整失败链（中文更常见）：

```text
用户点「中」
  → unlockSpeech（共享 Audio）
  → 请求中文源1（百度 gettts）
  → 网络慢 / 接口无响应 / 返回非音频
  → 3.5s～数秒内未进入 playing → 记「源1超时」
  → 换源2 再 play()
  → 若此时已不在用户手势链内，或 WebView 策略严：
        play() can only be initiated by a user gesture
  → 回退系统 speechSynthesis
  → 当前浏览器无该接口
  → 整段报错，中文彻底没声
```

英文往往在源1就 `playing` + `ended` 了，**根本不会走进超时换源**，所以你感觉「英文秒出、中文不行」。

补充：

1. **听写模式**整段念中文，每个词都打中文 TTS；失败率被放大。  
2. **微信内置浏览器**常无 `speechSynthesis`，中文失败后零兜底。  
3. 有道 `dictvoice?le=zh` 主要仍服务词典场景，**对任意中文释义并不保证**和英文词一样稳。

---

## 5. 已做缓解（代码）

见 `docs/bug-手机点击无声音.md`，与中英文差异相关的部分：

1. 全局**一个** `HTMLAudioElement`，手势内静音解锁，后续只改 `src`（减轻换源手势问题）。  
2. 未开始播放超时换源；已开始则等 `ended`。  
3. 错误信息合并展示，避免只显示「不支持系统语音合成」。  
4. 按钮 `pointerdown` / `touchstart` 同步 `unlockSpeech()`。  

这些让「换源被手势杀掉」的情况变少，但**不能改变「中文源比英文源慢且脆」的本质**。

## 6. 根因分类（便于以后排障）

| 编号 | 根因 | 主要影响 | 是否已缓解 |
|------|------|----------|------------|
| A | 中英文使用不同第三方 URL | 中文慢/失败率高 | 未从根上换源 |
| B | 手机无 speechSynthesis | 中文失败无本地兜底 | 无法在所有 WebView 解决 |
| C | 超时后 play 丢手势 | 中文超时后二次失败 | 共享 Audio 部分缓解 |
| D | 微信等内置浏览器限制 | 双端都可能差，中文更惨 | 文档建议系统浏览器 |
| E | 非官方 TTS 风控/变更 | 某天中文集体挂 | 需可配置多源或自建 |

## 7. 可选后续方案（未实现，按优先级）

1. **自建/代理中文 TTS**（Edge TTS、讯飞、Azure、浏览器服务端转发）  
   - 优点：稳定、可控语速  
   - 缺点：要后端或云密钥  

2. **词库预生成中文音频**（构建时或导入时生成 mp3）  
   - 优点：点「中」和点「英」一样秒出  
   - 缺点：体积与生成流程  
   - **状态：2026-07-16 起已搁置，当前不实现** → [`方案-中文音频预生成.md`](./方案-中文音频预生成.md)

3. **仅听写用「英文念词 + 屏显中文」**（产品折中）  
   - 优点：不依赖中文 TTS  
   - 缺点：改变「听中文默写英文」的学习方式  

4. **桌面/部分 Android 强制优先系统中文 voice**（有中文语音包时）  
   - 优点：免在线  
   - 缺点：iOS/微信仍可能没有  

5. **探测延迟，自动跳过慢源**  
   - 对首源做 1s 级探测，减少干等

## 8. 给使用者的临时说明

- 点「英」正常、「中」不行：**多半是中文在线接口慢或被拦**，不是单词数据坏了。  
- 听写依赖中文朗读：请用 **Safari/Chrome**，保证 **联网**，关静音、调媒体音量。  
- 勿用微信内置浏览器打开局域网调试页。  
- 开发：`npm run dev:host` 后手机强刷。
